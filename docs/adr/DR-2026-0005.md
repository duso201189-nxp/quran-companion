# DR-2026-0005 — Learning Engine Architecture

## Relationship to DR-2026-0003 and DR-2026-0004

This record **amends one specific point** of
[DR-2026-0003](DR-2026-0003-sprint8-data-architecture.md): that
record's `srs_cards` sketch stated it would eventually *replace* the
simple Revision Queue (`ayah_statuses.status='review'`, its own
Decision 2/B) once real spaced repetition was needed. Sprint 10 is
that moment, and Decision 1 below reverses that specific expectation
— Revision Queue is **not** replaced. Everything else in
`DR-2026-0003` (schema discipline, migration approach, Bookmark
Collections shape) and all of `DR-2026-0004` (streak canonical
source, Daily Goal storage split, Revision Queue reuse via
`UserContentRepository`) remain unchanged and in force. Neither
record's `status` changes.

## Context

Sprint 10's own architecture review (pre-implementation, full
codebase read against `ROADMAP.md`'s Bước 9 — "Flashcard SM-2 + Quiz
+ Thống kê + Nhật ký") found that treating Review, Scheduler, and
Quiz as three independent one-off features would repeat a mistake
this project had already made once and corrected: Sprint 8/9 computed
streak two different ways from two different sources before
`DR-2026-0004` forced a single canonical answer. Building a
spaced-repetition scheduler, a review-session UI, and a quiz system
without first deciding how they relate to each other and to the
Revision Queue Sprint 9 had just shipped risked the same kind of
divergence — this time between `ayah_statuses.status='review'` (the
existing, working Revision Queue) and whatever new "what needs
reviewing" concept a scheduler would introduce.

Three facts specifically shaped the decisions below:

1. `DATABASE.md`'s own "Đã định" (planned) tier already sketched
   `srs_cards` as *generic* (`item_type` + `item_id`, not hardcoded to
   one content kind) but *replacing* the Revision Queue — a framing
   nobody had revisited since it was written pre-Sprint-8.
2. `ARCHITECTURE.md` §10 had already decided SM-2 over FSRS ("đơn
   giản, đã kiểm chứng (Anki)"; FSRS explicitly deferred to v2+) —
   this sprint did not need to re-litigate the algorithm, only make
   sure the choice stays swappable.
3. Vocabulary data (`lemmas`/`word_instances`) does not exist in any
   form — no Drift table, no shipped content, no importer, no decided
   source (`TODO.md`, still open). Any feature depending on
   word-by-word content was therefore blocked on a prerequisite this
   sprint could not resolve.

## Options Considered

**1. Revision Queue vs. Scheduler relationship.**
*Option A — `srs_cards` replaces `ayah_statuses.status='review'`
outright*, per `DR-2026-0003`'s original sketch. Rejected: this
would force a data migration (backfilling scheduling defaults for
every already-flagged Ayah) and would remove a simple, working,
independently useful feature (quick manual flagging) in favor of a
more complex one, when both can coexist.
*Option B — the two stay fully independent, no relationship at all.*
Rejected: this reintroduces the exact "two disagreeing sources"
failure mode `DR-2026-0004` had to fix for streak — nothing would
guarantee `srs_cards` membership tracks what the user actually
flagged for review.
*Option C — Scheduler consumes Revision Queue as its membership
source, without owning or replacing it.* Chosen (Decision 1).

**2. Scheduling algorithm coupling.**
*Option A — call SM-2 math directly from the repository/provider,
no abstraction.* Rejected: `ARCHITECTURE.md` §10 already anticipates
an FSRS migration at v2+; hardcoding SM-2 would make that a rewrite
instead of a swap.
*Option B — a `SchedulingAlgorithm` interface, SM-2 as the only
implementation today.* Chosen (Decision 3) — costs one interface and
one small pure class, for a real future swap point.

**3. Quiz content strategy.**
*Option A — a dedicated question-bank table, authored or generated
once and stored.* Rejected: duplicates Qur'an content already fully
present in Group A, and requires its own authoring/maintenance
pipeline this sprint has no scope for.
*Option B — generate questions at runtime from Group A content,
persist only results.* Chosen (Decision 4) — zero content
duplication, zero new Group A schema.

**4. Flashcards this sprint.**
*Option A — build a minimal/temporary version against partial or
placeholder vocabulary data.* Rejected explicitly — the content
source itself is an unresolved product decision (`TODO.md`), and a
temporary data model would need to be thrown away once a real source
is picked.
*Option B — defer entirely, keep the domain generic (`item_type`
already supports `'lemma'`) so nothing architectural blocks it
later.* Chosen (Decision 4/scope).

## Decision

### Decision 1 — Revision Queue remains the business source of truth

`ayah_statuses.status='review'` (Revision Queue) is not replaced,
not deleted, and not reduced in scope. The Scheduler **consumes** the
Queue — it reads current membership and layers scheduling metadata
(`ease_factor`/`interval_days`/`repetitions`/`due_date`/`state`) on
top — but never becomes the thing a user interacts with to flag an
Ayah for review. The Revision Queue screen (`/revision-queue`,
Sprint 9) is untouched by this sprint.

### Decision 2 — Learning Engine architecture

An organizing concept, not a concrete class:

```
Learning Engine
├── Revision Queue
├── Scheduler
├── Quiz
├── Flashcards (future)
├── Hifz (future)
└── AI Tutor (future)
```

Everything under this umbrella that needs "what should the user study
next" logic depends on the `SchedulingAlgorithm` abstraction (Decision
3), not on any specific algorithm or on each other directly.

### Decision 3 — SchedulingAlgorithm abstraction

`SchedulingAlgorithm` (`lib/features/learning/domain/scheduling_algorithm.dart`)
is a pure-Dart interface. Current implementation: `SM2SchedulingAlgorithm`
(classic SuperMemo-2). Future: FSRS, or other algorithms, can be added
as a second implementation and swapped in via
`schedulingAlgorithmProvider`'s override — without changing
`SchedulerRepository`, any provider that depends on it, or any UI.

### Decision 4 — Quiz architecture

Questions are generated dynamically at session start
(`QuestionGenerator` implementations, pure Dart, no database
dependency) from Group A content (`QuranRepository`, via
`quizContentPoolProvider`). **No Question Bank.** Qur'an content
remains the single source of truth — no question text, no Ayah text,
is ever written to Group B. Only quiz results (`quiz_results`:
type/surah/score/total/timestamp) are persisted. Flashcards are
explicitly deferred — blocked on the unresolved word-by-word data
source, not on architecture; no temporary solution was built.

### Decision 5 — Layering

```
Domain
  ↓
Repository
  ↓
Provider
  ↓
Presentation
```

No layer may skip another: widgets read/act only through providers;
providers compose repositories (and, for Scheduler, pure domain
objects) but contain no scheduling or question-generation logic
themselves; repositories adapt persistence and depend on domain types
but never on Riverpod; domain code (`SchedulingAlgorithm`,
`QuestionGenerator`) has zero Flutter/Drift/Riverpod dependency.

### Decision 6 — Testing strategy

Every layer ships automated tests in the same phase that introduces
it — reversing Sprint 9's explicit "no tests per phase" scoping.
Domain logic (SM-2 math, the 4 question generators) gets pure unit
tests with no Flutter/Drift dependency; repositories get
Drift-in-memory tests; providers get `ProviderContainer` tests;
screens get widget tests.

## Consequences

- **Extensibility**: a new question type is one new
  `QuestionGenerator` implementation added to `QuizQuestionFactory`'s
  list — no existing generator or factory code changes. A future
  Flashcard or Hifz feature reuses `SchedulingAlgorithm`/
  `SchedulerRepository` unchanged by giving `item_type` a new value.
  An FSRS migration is a provider override, not a rewrite.
- **Maintainability**: Revision Queue, Scheduler, and Quiz each have
  exactly one owner of their own data (`UserContentRepository`,
  `SchedulerRepository`, `QuizRepository` respectively) — no table is
  owned by two repositories, avoiding the streak-style divergence
  this review was specifically watching for.
- **Clean architecture**: domain purity is enforced, not just
  documented — `SchedulingAlgorithm` and `QuestionGenerator` are
  independently testable with zero test-harness/database setup.
- **Reusable learning engine**: the Scheduler and its provider-layer
  orchestration (`schedulerSyncProvider`, `dueReviewCardsProvider`)
  are generic over `item_type`, ready for Hifz or Flashcards to plug
  into the same tables and repository without a schema change.
- Two additive schema migrations: `srs_cards` (`schemaVersion` 3→4)
  and `quiz_results` (`schemaVersion` 4→5) — neither backfills
  historical data (no migration-time computation of scheduling
  defaults for already-flagged Ayahs; the Scheduler self-heals on
  next read instead), matching this project's established migration
  discipline.
- Two intentional deviations from `DATABASE.md`'s original rough
  sketches, each documented at the point of definition: `srs_cards`
  uses `UNIQUE(item_type, item_id)` rather than
  `UNIQUE(user_id, item_type, item_id)` (matching every other table
  in `user_tables.dart`, since a nullable pre-login `user_id` would
  make the wider constraint a no-op); `quiz_results.surah_id` is
  nullable (a `'mixed'` quiz session spans many Surahs, not one).
- Reversibility: soft. Both schema additions are purely additive and
  reversible by a further migration if needed; the algorithm and
  question-type choices are both behind swappable abstractions.
